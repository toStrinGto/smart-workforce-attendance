# 智工考勤 — 产品需求与开发背景文档

## 文档目的

本文档面向后端开发团队，介绍"智工考勤"系统的业务背景、用户角色、核心功能流程及数据关系。帮助后端开发者理解**为什么做**、**为谁做**、**做什么**，以便更准确地实现 API。

配套文档：`BACKEND_API.md`（数据库设计 + 43 个 API 端点详细规范）

## 1. 项目背景

### 1.1 行业痛点

建筑施工行业的劳动力管理面临以下核心问题：

1. **考勤记录不透明** — 传统纸质签到易伪造、难追溯，工人对自身出勤记录缺乏知情权
2. **记工效率低** — 班组长每天需要逐个核对几十名工人的出勤情况，手动记录，耗时且易出错
3. **薪资纠纷频发** — 由于考勤记录不规范，工人与承包方之间经常因工时、加班、薪资计算产生争议
4. **管理者缺乏全局视野** — 老板/管理员无法实时掌握各项目的人力投入、成本支出、异常情况
5. **报销流程混乱** — 材料费、交通费等支出缺少规范的审批流程和凭证管理
6. **信息孤岛** — 合同、结算、考勤、报销分散在不同系统，无法形成完整的项目成本视图

### 1.2 产品定位

**智工考勤**是一款面向建筑施工行业的移动优先考勤管理系统，覆盖从工人打卡、班组长记工、管理者监控到财务核算的完整链路。

核心价值主张：
- **工人**：透明了解自己的出勤、加班、收入
- **班组长**：高效批量记工，快速处理考勤异常
- **老板**：实时掌控项目进度、人力成本、财务状况
- **管理员**：统一管理项目、员工、考勤数据

### 1.3 技术架构概览

## 2. 用户角色

系统定义了 4 种角色，每种角色有独立的操作界面和数据权限：

### 2.1 工人 (worker)

**典型用户**：建筑工地的一线工人（木工、钢筋工、泥瓦工、水电工、普工等）

**使用设备**：个人手机（移动端）

**核心诉求**：
- 每天上下班打卡，留下考勤凭证
- 查看自己本月的出勤日历（哪天正常、哪天迟到、哪天缺勤）
- 了解自己的加班时长和预计收入
- 提交报销申请

**工作场景**：工人到达工地后打开手机 APP 打卡上班，下班时再打一次卡。系统自动记录打卡时间和定位。工人可以在日历页面查看本月每天的打卡详情（上班/下班时间、打卡照片），在统计页面看到累计出勤天数、加班时长和预估收入。

### 2.2 班组长 (foreman)

**典型用户**：管理一个班组（10-30人）的一线管理者

**使用设备**：个人手机（移动端）

**核心诉求**：
- 高效地为班组内所有工人批量记录当天考勤
- 处理工人的考勤异常（如忘打卡、迟到）
- 了解当天现场出勤情况
- 提交报销申请和施工日报

**工作场景**：班组长每天下班前打开"批量记工"页面，选择当前项目，勾选今天出勤的工人，设置白班天数（0/0.5/1）和加班时长，一键提交所有工人的考勤记录。当系统自动检测到异常（工人忘记打卡、打卡时间异常），班组长在"异常处理"页面查看并逐个处理——可以补记出勤天数和加班时长（通过），或驳回并填写原因。

### 2.3 老板 (boss)

**典型用户**：施工企业老板或高层管理者

**使用设备**：个人手机（移动端）

**核心诉求**：
- 一目了然地看到企业经营全貌（收入合同、结算、收款、项目状态）
- 监控各项目的出勤率和人力投入
- 审批报销单
- 管理合同和收入结算
- 核算项目成本（人工成本 + 报销支出）

**工作场景**：老板打开 APP 首页即看到"企业看板"——收入合同总额、收入结算、收款金额、开票金额，以及活跃项目数、未收款金额、待还款等关键指标。点击各指标可进入详情页面。在"考勤"标签页可以查看所有员工的今日出勤状态、每个项目的出勤统计。在"审批"标签页可以审批或驳回报销申请。在"合同"和"结算"页面管理收入/支出合同及收入结算。

### 2.4 管理员 (admin)

**典型用户**：企业内勤或系统管理员

**使用设备**：PC 电脑（桌面端，240px 固定侧边栏布局）

**核心诉求**：
- 全面管理所有项目和员工信息
- 查看和修改考勤记录
- 配置系统参数（考勤规则、项目参数、通知设置）
- 导出数据

**工作场景**：管理员通过 PC 浏览器访问管理后台。左侧是固定侧边栏导航（首页、项目管理、员工管理、考勤管理、系统设置），右侧是主内容区。管理员可以新建/编辑/删除项目，管理员工日薪，查看所有考勤数据并修改异常记录，配置考勤规则等系统参数。

## 3. 核心业务流程

### 3.1 考勤数据流转（核心链路）

这是系统最重要的数据流，从数据采集到薪资核算的完整路径：

```
工人打卡 (punch)                    班组长记工 (batch attendance)
     │                                       │
     ▼                                       ▼
 ┌─────────────────────────────────────────────┐
 │           attendance_record 表               │
 │  worker_id | project_id | date | status     │
 │  clock_in | clock_out | day_shift           │
 │  overtime_hours | notes                     │
 └──────────┬──────────────┬───────────────────┘
            │              │
     ┌──────▼──────┐  ┌───▼────────────┐
     │  考勤异常    │  │  统计聚合       │
     │  exception   │  │  summary/stats │
     │  (系统自动    │  │                │
     │   生成)      │  │  ┌────────────┐│
     └──────┬──────┘  │  │ 老板看板   ││
            │         │  │ 管理员报表 ││
     ┌──────▼──────┐  │  └────────────┘│
     │ 班组长处理   │  └────────────────┘
     │ 通过/驳回    │
     │ 同步更新     │
     │ 考勤记录     │
     └─────────────┘
```

**详细步骤**：

1. **数据采集（两个入口）**：
   - 工人自助打卡：通过手机上传照片 + GPS 定位 + 时间戳
   - 班组长批量记工：选择项目 → 勾选工人 → 设置白班/加班 → 一键提交

2. **异常检测**：
   - 系统自动识别：未打卡、迟到（上班晚于 08:00）、早退、缺下班卡
   - 自动生成 exception 记录，状态为 `pending`

3. **异常处理**：
   - 班组长/管理员查看待处理异常列表
   - 通过：补记 dayShift + overtimeHours，同步更新 attendance_record
   - 驳回：填写驳回原因，exception 状态变为 `rejected`

4. **数据消费**：
   - 老板：查看各项目今日出勤率、近 7 天趋势、员工明细
   - 管理员：查看和修改任意考勤记录
   - 工人：查看自己的月度考勤日历和收入统计

### 3.2 报销审批流程

```
工人/班组长提交报销                    老板审批
      │                                  │
      ▼                                  ▼
  reimbursement 表                   PUT approve
  status: pending                    │
      │                    ┌─────────┴─────────┐
      │                    │                   │
      │               approved:true       approved:false
      │                    │                   │
      ▼                    ▼                   ▼
  等待审批            status: approved    status: rejected
                          │
                     打款完成后
                     status: paid
```

**详细步骤**：

1. 工人/班组长/老板自己都可以提交报销申请，填写：费用类型（材料费/交通费/餐饮费/住宿费/办公费/其他）、金额、事由、凭证照片（最多 9 张）
2. 报销提交后进入 `pending` 状态
3. 老板在"审批报销"页面看到待审批列表，可以逐条同意打款或驳回
4. 同意后状态变为 `approved`；打款完成后由管理员手动标记为 `paid`
5. 驳回后状态变为 `rejected`

### 3.3 合同与结算管理

```
合同 (contract)                     收入结算 (income_settlement)
  收入合同 ──INCOME──┐                  ┌── 记录实际到账
  支出合同 ──EXPENSE──┤                  │
                      ▼                  ▼
              项目成本核算 (project-cost)
              人工成本 = Σ(工人出勤天数 × 日薪 + 加班工时 ÷ 6 × 日薪)
              报销成本 = Σ(项目相关报销金额)
              总成本 = 人工成本 + 报销成本
```

**业务逻辑**：
- 老板管理收入合同和支出合同，记录签约方、金额、日期
- 合同是高层面的财务承诺，收入结算则是具体的到账记录
- 收入结算关联项目和账期，含税前金额、税率、含税总额
- 项目成本由人工成本（考勤数据）和报销支出自动计算

### 3.4 日报流程

```
班组长/工人选择模板 → 填写日报内容 → 上传施工照片 → 提交
                                                        │
                                              老板收到日报
                                              标记已阅/未阅
```

日报模板（daily_report_template）可由班组长/老板创建和编辑，提供固定格式（如"施工内容/人员情况/安全情况/明日计划"），方便快速填写。提交后的日报在历史列表中显示，老板可以查看并标记已阅。

## 5. 关键数据实体关系

### 5.1 核心实体

```
user (用户)
 ├── role: worker → 有 dailyWage（日薪）
 ├── role: foreman → 管理一组 project_worker
 ├── role: boss → 查看所有数据，审批报销
 └── role: admin → 全部权限

project (项目)
 ├── status: 未开工 → 施工中 → 维保中 → 已完工
 ├── manager: 项目经理姓名
 ├── progress: 进度百分比 (0-100)
 └── budget: 项目预算

project_worker (项目-工人关联)
 ├── 一个工人可属于多个项目（在不同班组）
 ├── team: 班组名称（木工班组、钢筋班组等）
 └── role: 工种（木工、钢筋工、泥瓦工、水电工、普工）

attendance_record (考勤记录)
 ├── 核心业务数据，每人每天每项目一条
 ├── day_shift: 出勤天数 (0 / 0.5 / 1)
 ├── overtime_hours: 加班时长 (小时)
 └── 唯一约束: (worker_id, project_id, date)
```

### 5.2 关键查询场景

后端需要高效支持以下查询：

| 查询场景 | 涉及表 | 频率 | 建议 |
|---------|--------|------|------|
| 今日全公司出勤统计 | attendance_record + project | 高（每次打开管理员/老板首页） | Redis 缓存，每日更新 |
| 某项目近 7 天出勤趋势 | attendance_record | 高 | 按项目+日期范围索引 |
| 某工人月度考勤日历 | attendance_record | 中 | 按 worker_id + 月份范围 |
| 待处理异常列表 | exception | 高 | 按 status='pending' 索引 |
| 某项目人工成本 | attendance_record + user(dailyWage) | 中 | 按 project_id 聚合 |
| 老板仪表盘汇总数据 | contract + income_settlement + project + reimbursement | 高 | Redis 缓存，定时刷新 |

### 5.3 考勤状态机

```
                 班组长记工
                    │
     ┌──────────────┼──────────────┐
     │              │              │
  dayShift=0    dayShift>0     dayShift>0
     │         + overtime=0   + overtime>0
     ▼              ▼              ▼
  absent        present       present
                              (标记加班)


              工人打卡
                 │
     ┌───────────┼───────────┐
     │           │           │
  上班正常    上班迟到     未打上班卡
  下班正常    下班早退     未打下班卡
     │           │           │
     ▼           ▼           ▼
  normal     abnormal      missing
  (或 overtime)  (迟到/早退)  (缺卡)
                    │
                    ▼
              自动生成 exception
              status: pending
                    │
            ┌───────┴───────┐
            │               │
         班组长通过       班组长驳回
            │               │
            ▼               ▼
  更新 attendance_record  status: rejected
  status: handled
```

## 6. 班组与工种体系

建筑施工行业的组织结构是：

```
公司
 └── 项目 (project) — 如"绿地中心二期"
      └── 班组 (team) — 如"木工班组"、"钢筋班组"
           └── 工人 (worker) — 如张三（木工）、李四（钢筋工）
```

- **班组**：由一名班组长（foreman）和若干工人组成，按工种划分
- **一个工人可以在多个项目工作**，但在一个项目中只属于一个班组
- **班组长也是用户**，role='foreman'，不直接出现在考勤记录中（班组长自己也可能有考勤记录）

前端中班组信息存储在 `project_worker` 关联表的 `team` 字段，工种存储在 `role` 字段。

## 7. 日薪与收入计算

### 7.1 工人日薪

每个工人有一个 `dailyWage`（日薪），单位为元/天。管理员可以在"员工管理"页面内联编辑。

### 7.2 收入计算公式

```
月收入 = (正常出勤天数 × 日薪) + (加班天数 × 日薪)
```

- 正常出勤天数：该月 `day_shift > 0` 的记录总天数
- 加班天数：该月总加班工时 ÷ 6（行业惯例：6 小时算一个工日）
- 加班工时：该月所有记录的 `overtime_hours` 之和

### 7.3 项目成本计算

```
项目人工成本 = Σ 每个工人的(出勤天数 × 日薪 + 加班工时 ÷ 6 × 日薪)
项目报销成本 = Σ 该项目关联的所有已批准报销金额
项目总成本 = 人工成本 + 报销成本
```

## 8. 移动端 vs 桌面端差异

### 8.1 移动端（worker / foreman / boss）

- 底部 4 个 Tab 导航（角色不同 Tab 不同）
- 橙色主题色（#f97316 orange-500）
- 大字体、大按钮，适合工地环境操作
- 使用 `pt-12`（顶部状态栏）和 `pb-20`（底部 Tab 栏）的安全区域

### 8.2 桌面端（admin）

- 左侧固定 240px 侧边栏 + 右侧主内容区
- 中性灰蓝色调
- 表格、表单、Dialog 为主的交互方式
- 面包屑导航
- 支持键盘快捷键（Enter 保存、Escape 取消）

## 9. 系统设置项

管理员在"系统设置"页面可配置以下参数（前端已完成 UI，后端需新建 `system_setting` 表）：

### 9.1 考勤规则
- 上班时间（默认 08:00）
- 下班时间（默认 18:00）
- 迟到宽限分钟数（默认 15 分钟）
- 加班起算时间（默认 18:30）
- 工时计算方式（按天 / 按小时）

### 9.2 项目参数
- 预算单位（万元 / 元）
- 默认项目状态
- 进度阈值提醒
- 项目编号前缀

### 9.3 通知设置
- 考勤异常提醒（开关）
- 周报自动生成（开关）
- 项目延期预警（开关）
- 每日考勤汇总（开关）

### 9.4 数据管理
- 导出考勤数据
- 导出项目数据
- 同步数据
- 清除缓存

## 11. 前后端对接流程

### 11.1 开发环境配置

前端通过环境变量控制 API 来源：

```env
# .env 文件
VITE_MOCK_ENABLED=false          # 关闭 mock 模式
VITE_API_BASE_URL=http://localhost:8080/api/v1   # 指向后端地址
```

关闭 mock 后，前端所有请求将直接发往后端服务。

### 11.2 CORS 配置

后端需要配置跨域，允许前端开发服务器访问：

```
Access-Control-Allow-Origin: http://localhost:3000
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS
Access-Control-Allow-Headers: Content-Type, Authorization
```

### 11.3 对接验证步骤

建议按以下顺序逐步对接：

1. **认证流程** — 先跑通 login → 拿到 token → refresh token
2. **项目 CRUD** — 管理员端最基础的功能
3. **工人列表** — 确保基础数据查询正常
4. **批量记工** — 班组长核心流程
5. **考勤查询** — 各角色查看考勤数据
6. **异常处理** — 班组长/管理员处理异常
7. **报销流程** — 提交 + 审批
8. **合同与结算** — 老板端财务管理
9. **日报** — 模板 + 提交 + 历史
10. **仪表盘统计** — 确保聚合数据正确

## 12. 术语表

| 术语 | 英文 | 说明 |
|------|------|------|
| 班组 | team | 按工种划分的工作小组，由一名班组长管理 |
| 工种 | trade/role | 工人的技能分类：木工、钢筋工、泥瓦工、水电工、普工 |
| 白班 | day shift | 正常出勤天数，取值 0 / 0.5 / 1 |
| 加班 | overtime | 超出正常工时的工作时间，单位小时 |
| 记工 | attendance recording | 班组长为工人记录出勤的动作 |
| 打卡 | punch/clock-in/out | 工人通过手机 APP 记录上下班时间 |
| 考勤异常 | exception | 系统自动检测到的异常打卡行为（迟到、缺卡等） |
| 日薪 | daily wage | 工人每天的工资标准（元/天） |
| 进度款 | progress payment | 按工程进度支付的款项 |
| 结算款 | settlement payment | 工程完工后的最终结算款项 |
| 施工日报 | daily report | 每日施工情况的书面记录 |
